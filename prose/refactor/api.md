# API Layer Refactor Audit

Scope: `src/routes/api/**` and `src/lib/server/utils/**`.

Organized by theme. Each theme groups related findings so the refactor can be
tackled as a single change rather than a scattered set of edits.

---

## Cron auth duplication ✅ RESOLVED (`7407373`, 2026-04-22)

**Category:** redundant · **Leverage:** high

**Location:**
- `src/routes/api/cron/seed-templates/+server.ts:9-24`
- `src/routes/api/cron/mark-template-official/+server.ts:16-29`
- `src/routes/api/cron/repair-template/+server.ts:24-37`

**What.** All three cron routes open with the same ~13-line block: check that
`env.CRON_SECRET` is configured, read the `authorization` header, build the
expected `Bearer …` string, compare with `timingSafeEqual` using `Buffer.from`,
and return `{ error: 'unauthorized' }` on any failure.

**Why it smells.** This is the most sensitive code in the cron surface and the
hardest to review when it drifts. A length-mismatch check is required by
`timingSafeEqual`, and a regression in one copy (e.g. a missing length guard)
would silently create a timing oracle. The duplication also prevents a single
test from covering the contract.

**Direction.** Extract `requireCronAuth(request: Request): Response | null` (or
a `throw`-style helper) into `src/lib/server/utils/auth.ts`. Each cron route
becomes one line. Keep the helper colocated with the other auth utilities so
the timing-safe comparison has one owner.

**Resolution:** `requireCronAuth(request: Request): Response | null` exported
from `src/lib/server/utils/auth.ts`; all three cron routes call it. The three
original copies were behaviorally identical. Note: a missing `CRON_SECRET`
still returns 401 (matching prior behavior) rather than 500 — preserved
intentionally as part of "don't churn the response shape here"; open for a
follow-up when the response-contract drift finding lands.

## UUID validation fragmentation ✅ RESOLVED (`07e4986`, 2026-04-22)

**Category:** redundant · **Leverage:** high

**Location:**
- `src/lib/server/utils/api.ts:15` — `UUID_REGEX` (loose: `[45]` version, `[89ab]` variant)
- `src/routes/api/cron/mark-template-official/+server.ts:7-8` — second regex (stricter: `[1-5]`, `[89abAB]`)
- `src/routes/api/cron/repair-template/+server.ts:7-8` — third regex (identical to the cron one above)

**What.** Three separate UUID regex literals for the same concept. The utility
in `api.ts` accepts versions 4 and 5; the two cron copies accept versions 1–5.
Neither the utility nor the cron routes document why they differ.

**Why it smells.** Validators diverging across the codebase make it easy to
accept an id in one place and reject it in another. Two of the three copies
exist because nothing from `utils/api.ts` is being reused.

**Direction.** ~~Decide on the canonical shape (almost certainly v4-only, since
every id in the system is generated by `randomUUID()`) and export it as a
single `UUID_REGEX` constant plus `isValidUUID` from `utils/api.ts`.~~
Accept v4 + v5: documents and user-created templates use `crypto.randomUUID()`
(v4), but **official templates are v5**, derived deterministically from their
manifest id via `uuidv5()` in
`src/lib/server/services/templates/seed.ts:29`. The original "v4 or v5"
regex in `utils/api.ts` was correct for this reason. Have the cron routes
import the shared `isValidUUID`.

**Resolution:** Consolidated around the existing v4+v5 `UUID_REGEX` in
`utils/api.ts`. Both cron routes now import `isValidUUID` from the shared
helper. (An initial pass tightened to v4-only per a mistaken reading of the
audit — official template ids then failed validation at runtime. Reverted.)

## Pervasive UUID-param boilerplate

**Category:** redundant · **Leverage:** medium

**Location:** ~23 invocations of `validateUUID()` across route handlers, most
following this exact three-line shape:

```ts
const id = event.params.id;
const invalid = validateUUID(id);
if (invalid) return invalid;
```

e.g. `src/routes/api/templates/[id]/+server.ts:22-25,58-62,74-78`,
`src/routes/api/templates/[id]/content/+server.ts:13-17`, and similar in the
import/star/reset endpoints.

**What.** `validateUUID()` returns a `Response | undefined`, so every caller
restates the same early-return pattern.

**Direction.** Either
- change the helper to throw, matching the style used by `requireAuth()` /
  `errorResponse()` (the handler's surrounding `try/catch` + `handleServiceError`
  already expect thrown responses); or
- introduce `requireUuidParam(event, name)` that returns the validated string
  or throws. Keep the ergonomics of one line per id.

The `paramName = 'id'` default parameter on `validateUUID` can be removed: in
practice every call site either uses the default or passes `'document_id'`.

## Response contract drift

**Category:** smelly · **Leverage:** high

**Success payload inconsistency.** Success responses use at least three
different shapes across the API:

- `{ success: true }` — `api/documents/[id]/+server.ts:74`,
  `api/templates/[id]/+server.ts:79`, `api/templates/[id]/reset/+server.ts:22`
- `{ ok: true }` — `api/beta-program/+server.ts:27`
- `{ success: true, … }` with extra fields — the three cron routes
- HTTP 204 no-body — `api/metrics/export/+server.ts`

**Error payload inconsistency.** Two shapes coexist:

- Flat: `{ error: 'code', message: '…' }` (the standard, produced by
  `errorResponse()` in `utils/api.ts`).
- Nested: `{ error: { code, message, fields } }` — feedback defines a local
  `errorBody()` helper in `src/routes/api/feedback/+server.ts:17-39`.
- Minimal: `{ error: 'unauthorized' }` — the three cron routes bypass
  `errorResponse()` entirely.

**HTTP status drift.** Creates correctly use 201 (`api/documents/+server.ts:26`,
`api/templates/+server.ts:71`). Void operations (DELETE, PUT reset) return 200
with a `{ success: true }` wrapper rather than 204, except for
`metrics/export/+server.ts` which correctly returns 204.

**Why it smells.** Clients cannot rely on a single schema; each endpoint
teaches them a new one. The feedback endpoint's nested error shape is the
outlier that will trip tooling expecting the flat shape.

**Direction.**
1. Extend `errorResponse()` (or add a sibling) to carry optional
   field-level error detail, then have feedback call it.
2. Pick one success convention. A sensible rule:
   - Creates → 201, body is the created resource.
   - Updates returning data → 200, body is the resource.
   - Void operations (DELETE, reset, fire-and-forget record) → 204, no body.
   - Remove `{ success: true }` and `{ ok: true }` wrappers entirely.
3. Cron routes should call `errorResponse()` too — the `{ error: 'code' }`
   shape is inconsistent with everything else.

## Request parsing abstractions with single callers

**Category:** over-engineered · **Leverage:** low

**Location:** `src/lib/server/utils/request-schemas.ts` exports
`parseCreateTemplateRequest`, `parseUpdateTemplateRequest`,
`parseCreateDocumentRequest`, `parseUpdateDocumentRequest`. Each parser is
imported from exactly one route.

**Why it smells.** The module is doing an interface's worth of ceremony
(imports, named exports, shared types) to move five-line validators one file
away from their only caller. The savings (route handler stays short) are real
but modest; the cost (reader must jump files to see validation rules) is not.

**Direction.** Two reasonable options:
- Leave as-is but add a short header comment explaining that this file is the
  single-source-of-truth for request-body schemas and future endpoints should
  add their parser here.
- Migrate to a schema library (Zod/Valibot) so the validators compose and the
  types derive from the schema. This is the higher-leverage move if any
  validator ever needs to grow.

Do not keep the current shape *and* inline new validators — pick a side.

## `cdn-cache.ts` used by exactly one handler

**Category:** over-engineered · **Leverage:** low

**Location:** `src/lib/server/utils/cdn-cache.ts` (six exports); only
`src/routes/api/templates/[id]/+server.ts:14-18,33-39` imports it.

**Why it smells.** A utility module with a `CacheProfile` type, a profile
constant, an ETag builder, and a header-applier for one consumer is speculative
infrastructure. No other route caches through it, and the CDN/browser/SWR
tuning is template-detail-specific.

**Direction.** Either inline back into the template detail handler (cheapest,
consistent with "three similar lines is better than a premature abstraction"),
or use it from at least one more route (public documents is a natural second
candidate) before keeping the abstraction.

## `ambiguous-response-logging.ts` tangential logging utility

**Category:** over-engineered / smelly · **Leverage:** low

**Location:** `src/lib/server/utils/ambiguous-response-logging.ts`, called
three times total (`api/public/documents/[id]/+server.ts:24-29,39-45`;
`api/metrics/export/+server.ts:46-51`).

**Why it smells.** The helper wraps an "ambiguous-response" concept (used when
a 404 could be privacy-preserving vs a true miss) with a correlation id and
optional user/resource context. The abstraction isn't pulling its weight at
three call sites, and the concept isn't documented anywhere in `prose/designs`.

**Direction.** Either document the semantics (what counts as "ambiguous", why
we log it, what the correlation id enables) in
`prose/designs/ERROR_SYSTEM.md`, or collapse it into plain `console.warn` with
the correlation id inline. Decide whether this logging is load-bearing first.

## Ad-hoc `console.error` in feedback route

**Category:** smelly · **Leverage:** medium

**Location:** `src/routes/api/feedback/+server.ts` — 10+ `console.error` calls.

**What.** Each external-API failure branch logs with a different message; some
include the full response object.

**Why it smells.**
- Logs are unstructured, so grepping for the feedback flow requires knowing
  the exact message phrasing.
- A few calls serialize whole response bodies from GitHub/GitLab, which can
  include rate-limit headers, urls, and occasionally payload fragments.

**Direction.** Pick a small logger (console with a prefix is fine) and a
consistent shape: `{ event, error, context }`. Treat structured logging as a
cross-cutting concern — apply the same rule in the services layer
(see [services.md § Console logging](./services.md#console-logging)).

## Optional JSON body parsing heuristic

**Category:** smelly · **Leverage:** low

**Location:** `src/routes/api/templates/[id]/import/+server.ts:23-40`.

**What.** Decides whether to parse a body by testing
`event.request.headers.get('content-length') !== '0' && event.request.body`.

**Why it smells.** `content-length` can legitimately be absent (chunked
encoding). `event.request.body` being present is unreliable with SvelteKit's
request object. A missing body will then throw inside `json()` and get caught
generically.

**Direction.** Add `parseOptionalJsonBody(request)` to `utils/api.ts` that
attempts `await request.json()` and returns `null` on `SyntaxError` or an
empty body. One helper, one convention. Apply in `import/+server.ts` and any
future endpoints with optional payloads.

## Redundant service-factory calls per request

**Category:** redundant · **Leverage:** low

**Location:** e.g. `src/routes/api/templates/[id]/+server.ts:27,59,76` — three
different handlers in the same file each call `getTemplateLibraryService()`.
24 calls total across the codebase.

**What.** The factory caches a singleton, so the cost is negligible. The noise
isn't: every handler opens with the same two lines of plumbing.

**Direction.** Once the provider/factory indirection is removed (see
[services.md § Provider indirection](./services.md#provider-indirection)),
every route imports the singleton directly and this goes away.

## Minor stylistic drift

**Category:** smelly · **Leverage:** low (aggregate)

These are cheap to fix if touched during any of the above refactors; not worth
a dedicated PR.

- **URL access.** Most routes use `event.url`; `api/documents/+server.ts:35`
  constructs `new URL(event.request.url)`. Standardize on `event.url`.
- **Parameter destructuring.** Some handlers do `async (event) => { … }`;
  others do `async ({ request, url }) => { … }`. Pick one.
- **`await guestAwareAuth(event)` with discarded result**
  (`src/routes/api/templates/+server.ts:19`). If the user is irrelevant, the
  call is a pure side-effect check; capture it anyway so future changes can
  thread the user through.
- **Auth flow via throw vs return.** `requireAuth` throws a `Response`;
  `validateUUID` returns one. Handlers need both patterns. Pick one to move to
  (throwing matches the `handleServiceError` centralization) and migrate the
  other.
- **`parseDryRunParam` in `repair-template/+server.ts:10-16`.** Seven lines to
  parse `"true"` into `true`. Collapse to
  `raw?.trim().toLowerCase() === 'true'`.
